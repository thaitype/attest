# Grill: Attest Spec V1

## Design Clarification (fundamental)

**CLI-as-canonical-interface.** The final goal is full automation. Agents are first-class callers of CLI commands. "CLI is the only writer" means: CLI is the canonical interface for state transitions, enforced by prompt/skill convention, not OS-level access control. Direct frontmatter edits are unsupported — they bypass state machine integrity but are not technically blocked. Git is the audit log. This reframes Q6, Q18, and any "security-relevant" framing that assumed agents need to be gated by humans.

## Resolved

Q4: What stops parallel agents from double-claiming an item? → Multi-agent supported from v0.1. Lock file via atomic rename per item folder. Spec amendments: (1) replace Open Q2 lean with lock file decision, (2) remove "Does not coordinate parallel work" from "What Attest does not do", (3) add conflict-error path to agent loop contract step 3. Why: item-per-folder layout supports .claim file naturally; atomic rename is POSIX-safe on local fs.
Q4b: Atomicity mechanism → lock file (atomic rename). Resolved via Q4.
Q5: Who populates `implementation_hints`? → Spec author (human or proposer agent) — hints are intent, written before implementation. They drive `attest affected` for TIA-lite. Why: hints live in spec.md alongside acceptance criteria; pre-implementation intent, not post-hoc discovery. Keeping spec.md stable after writing avoids blurring the spec-is-stable assumption.
Q7: What should `attest coverage` output? → (C) Both: ratio as summary line + item-level traceability matrix as detail. With `--format json`, matrix lets agents find gaps autonomously (which items have ≥1 test declaration, which are verified). Why: bare ratio is unactionable for an agent; matrix is achievable without body parsing (no criterion-level traceability needed in v0.1). Spec amendment: clarify "RTM-style" means item-level matrix, not criterion-level.
Q6: Should `attest review` verdict be writable by the reviewer agent or human-only? → Reviewer agent calls `attest mark-reviewed FR-001 --verdict pass` (a CLI command). CLI writes; git is the audit trail. Why: CLI is an interface for efficiency/consistency, not a security gate — full automation is the goal. Open Q18 resolved: git = audit log, no human gate required. Spec amendments: add `attest mark-reviewed` to CLI surface; add Pass 3 → `verified` transition to state machine; reframe "CLI is the only writer" in design principles to clarify it means interface consistency, not agent restriction.
Q2 (reopened): Does `attest done` get gated by a scan, and where in config? → Defer scan-on-done gate to future work. v0.1: `attest done` records the claim only; scan stays strictly post-`done` (loop step 6). `done` keeps "not yet checked" semantics → resolves the verifier conflict (no scan-gating of `done`). Namespace question for the gate also deferred. Why: the gap between `done` (claim) and scan/verify (reality) is Attest's core evidence — gating would suppress the false-claim signal and risk blocking on scan false-negatives. No spec change needed (loop already lists scan as optional post-done). Minor pre-existing note left as-is for v0.1: `verify.gates.require_test_link/require_commit` actually gate the `done` transition, so the `verify.gates` name is mildly misleading.
Q3: Should `stale` be auto-recoverable via `attest verify`, or require human intervention? → Configurable. Default: stale items excluded from `--ready`. Re-verify auto-transitions to `verified`/`failing`. Why: agents run autonomously overnight. **Spec amendments required:** (1) add `stale → verified | failing` arc to state diagram, (2) update `--ready` definition to explicitly exclude stale items, (3) add `verify.auto_recover_stale` config key (or similar) to config section. How to configure the per-drift-type behavior is Q3b (open). Verifier flagged conflict: scan is post-done by design in agent loop contract; gating `done` with scan contradicts "not yet checked" definition; config namespace unclear (`verify.gates` vs `transitions.done`).

Q3b: How should `stale` recovery be configured? → (3) trimmed: single `stale` section with two booleans for v0.1 — `stale.block_ready` (default true, hides stale from `--ready` regardless of cause) and `stale.auto_recover` (default true). Hard-coded rule (not a knob in v0.1): `auto_recover` applies to **code/test drift only**; **spec drift is never auto-recovered by `verify`** — it requires `attest accept-drift --spec` or a Pass-3 review before returning to `verified`. Per-drift-type config deferred to future work. Why: if only `spec.md` changed, tests still encode old behavior, so letting `verify` re-stamp `verified_against.spec` with the new hash would be a false `verified` (integrity hole). Spec amendments: (1) add `stale` config section; (2) document the spec-drift carve-out + refine Q3's blanket "re-verify auto-transitions"; (3) add `attest accept-drift FR-001 --spec` to CLI surface — promotes spec Open-Q12 lean to decided.

Q8: How does `scan` attribute a test name to the right item when names collide? → (C) loud-failure + (A) tie-breaker. Keep matching global, cheap, case-insensitive substring (per existing config), but add an `ambiguous` category to `scan_result` (alongside found/missing/orphan): a declared name matched by >1 item is reported `ambiguous`, never silently attributed. When `test_hints` are present, use them to disambiguate (tie-breaker); do NOT hard-scope scan to `test_hints` (avoids a name living outside its hints silently reading as "missing"). Reject (B) in-code `deattest()` macro — invasive, framework-coupled, violates the framework-agnostic/no-AST principle; deferred to future. Why: collisions become loud rather than wrong, preserving "your test name should just match" usability and feeding accurate coverage matrices (Q7). Spec amendments: (1) add `ambiguous` field to `scan_result` frontmatter + scan reporting; (2) document collision/tie-breaker behavior in Pass 1; (3) resolves spec Open-Q4 (test name resolution).

Q9: When does the `.claim` lock release, and how are stale/orphaned locks recovered? → (B) lease + TTL. `.claim` file shape: `{ agent, token, started_at }` (token = random id for conflict detection). Auto-released on any transition out of `in_progress` (`done`/`block`/`skip`/`reset`) — happy path. Stale recovery: `lock.ttl` config (default `"15m"`); `attest start` reclaims an expired claim automatically with a new token and logs the takeover. Displaced agent's later `attest done` is rejected via token mismatch (conflict-error path at step 5, mirror of Q4's step-3 path). `attest reset` always force-releases regardless of TTL. Lock is advisory — Attest cannot physically prevent a renegade agent from writing files; that's reeve's job. Deferred to future: heartbeat/renewal, pid/host liveness checks. Why: Q1/Q3 pin the use case as autonomous overnight agents — without TTL, one crash blocks an item until human intervention, breaking the unattended premise. 15m is ~1.5× typical atomic-item duration, generous enough that legit work finishes inside the lease. Spec amendments: (1) add `lock` config section with `ttl: "15m"` default; (2) document `.claim` file shape and lifecycle in Multi-agent / state machine section; (3) add agent loop contract step 4.5 — "if elapsed since `start` approaches `lock.ttl`, re-verify claim ownership before writing more files; bail to step 1 if token no longer matches"; (4) document conflict-error path at step 5 done; (5) explicitly note lock is advisory.

Q10: What does `attest review FR-001` package and emit? → (C) both formats, code referenced not embedded. Default JSON (per Q1 agent-as-consumer principle), `--format markdown` opt-in for human/council artifacts. Same payload contents both formats: item metadata (id, flow, status, spec/test hashes, verified_against), `spec.md` body embedded, `test.md` body embedded, flow `overview.md` body if applicable, `scan_result` + `verification_result` (Pass 1/2 evidence), `depends_on` items (id + status only), `implementation_hints` as paths + current commit SHA (NOT file contents embedded). Output to stdout for piping/composition. Why not embed code: files can be large, reviewer has its own context budget, and path+commit lets the reviewer fetch precise excerpts as needed — consistent with framework-agnostic/no-AST principle. Why both formats: Q1 establishes agents as primary consumer (JSON), but markdown bundles compose into chief/council artifacts for human review. Deferred to future: `--embed-code` flag for remote reviewers without filesystem access. Spec amendments: (1) define review payload schema (or canonical field list); (2) add `--format json|markdown` flag to `attest review` (default json); (3) document path-referenced code policy; (4) resolves spec Open-Q10.

Q11: When does `attest mark-reviewed --verdict pass` clear spec drift? → (C) Reading A with hash-guard. `mark-reviewed pass` clears spec drift and re-stamps `verified_against.spec`, but the verdict must reference the spec hash the reviewer was shown (already in the Q10 review payload). Clearing only happens if that hash equals the current `spec.md` hash; mismatch → verdict rejected with "stale payload — re-run review." `accept-drift --spec` remains as the shallow path (no review, human consent only) — two valid arcs out of spec-drift stale, both deliberate writes. Why: keeps the Pass-3 flow end-to-end (no orphan stale items after a clean review) while preserving Q3b's integrity guarantee at the state-machine level rather than by convention. Resolves the Q3b↔Q6 ambiguity flagged by the initial Final Review. Spec amendments: (1) `attest mark-reviewed FR-001 --verdict pass --against-spec sha256:abc` (or equivalent) signature; (2) document the hash-guard rule and stale-payload rejection; (3) state machine: list both `mark-reviewed (hash-match)` and `accept-drift --spec` as the two paths from spec-drift stale → verified; (4) document `accept-drift --spec` as the shallow consent path.
